Ticket Detail — 3 Layout Approaches

The core problem: "Identify fix" and "Public reply" appear twice — once in the quick strip, once as a full section at the bottom. Pick the structure that best solves it.

A

Unified AI Panel — one place to act

Remove the duplicate bottom section. One consolidated AI panel lives just below the header. All actions (fix, internal note, public reply) target the same output area. Persona selector always visible.

Pros

  • No duplication
  • AI actions close to context
  • Persona always accessible

Cons

  • Panel can grow tall before description
  • Still interrupts read flow slightly

B

Read-first — AI in the sidebar

Main column is pure reading: description → assets → thread. Right sidebar gets two panels: related issues on top, AI actions panel below. The engineer reads first, acts from the side.

Pros

  • Clean read flow — no interruptions
  • Context (similar issues) sits next to AI actions naturally
  • Best separation of concerns

Cons

  • Sidebar gets dense
  • AI output area is narrow
  • May need horizontal scroll on smaller screens

C

Tab split — "Read" vs "Act"

Two tabs: Ticket (description, assets, thread) and AI Copilot (fix identification, draft composition, similar issues all in one view). No sidebar — full width for each mode.

Pros

  • Maximum width for both content and AI output
  • AI tab can show richer output (edit system prompt, etc)

Cons

  • Context switching cost — can't see ticket & AI at once
  • Related issues buried in AI tab
